## 🔗 Key Polymorphic Relationships

### What is a Polymorphic Relationship?

A polymorphic relationship allows a model to belong to multiple other models on a single association. In this database, we use polymorphic relationships to avoid duplicate tables.

### 1. **Cart Items** (Products OR Services)

**Table:** `cart`
- `item_type` = `'App\Models\Product'` OR `'App\Models\Service'`
- `item_id` = ID of product or service

**Why Polymorphic?**
- Allows mixed cart (products + services in same order)
- Single cart table instead of `product_cart` + `service_cart`
- Unified checkout experience

**Example:**
```
Cart Item 1: item_type='App\Models\Product', item_id=5  → iPhone 15 Pro
Cart Item 2: item_type='App\Models\Service', item_id=12 → Web Development Consultation
```

---

### 2. **Order Items** (Products OR Services)

**Table:** `order_items`
- `item_type` = `'App\Models\Product'` OR `'App\Models\Service'`
- `item_id` = ID of product or service

**Why Polymorphic?**
- Same reason as cart (mixed orders)
- Price snapshot stored in `snapshot` JSON field

**Example Order:**
```json
{
  "order_id": 42,
  "items": [
    {"item_type": "Product", "item_id": 5, "snapshot": {"name": "iPhone 15 Pro", "price": 999.99}},
    {"item_type": "Service", "item_id": 12, "snapshot": {"name": "Web Development", "price": 150.00}}
  ]
}
```

---

### 3. **Reviews** (Products OR Services)

**Table:** `reviews`
- `reviewable_type` = `'App\Models\Product'` OR `'App\Models\Service'`
- `reviewable_id` = ID of product or service

**Why Polymorphic?**
- Unified review system for both products and services
- Same moderation workflow
- Shared rating logic

**Example:**
```
Review 1: reviewable_type='App\Models\Product', reviewable_id=5  → 5-star review for iPhone
Review 2: reviewable_type='App\Models\Service', reviewable_id=12 → 4-star review for Web Dev
```

## 📐 Database Design Patterns

### 1. **Hierarchical Categories** (Self-Referencing)

**Tables:** `product_categories`, `service_categories`

**Pattern:**
```
parent_id → categories.id (self-referencing foreign key)
```

**Example Structure:**
```
Electronics & Technology (parent_id = NULL)
├── Computers & Laptops (parent_id = 1)
│   ├── Desktop Computers (parent_id = 2)
│   │   └── Gaming PCs (parent_id = 3)
│   └── Laptop Computers (parent_id = 2)
└── Mobile Devices (parent_id = 1)
```

**Features:**
- Unlimited nesting depth
- Recursive count queries (parent shows total of all descendants)
- Circular reference prevention in model boot method

---

### 2. **Many-to-Many with Pivot Data**

**Tables:** `product_product_category`, `service_service_category`, `provider_service`

**Pattern:**
```
Pivot table with additional metadata:
- is_primary (boolean) - Mark one category as primary
- display_order (integer) - Sort order
- custom_price (decimal) - Provider-specific pricing
```

**Example:**
```
Product #5 → Categories:
  - Electronics (is_primary=1, display_order=1)
  - Featured Products (is_primary=0, display_order=2)
  - New Arrivals (is_primary=0, display_order=3)
```

---

### 3. **Soft Deletes**

**Tables:** `products`, `services`, `blog_posts`

**Pattern:**
```sql
deleted_at timestamp NULL
```

**Behavior:**
- Deleted records remain in database with `deleted_at` timestamp
- Queries exclude soft-deleted by default
- Can be restored or permanently deleted
- Preserves order history and relationships

---

### 4. **Price Snapshots**

**Table:** `order_items`

**Pattern:**
```json
snapshot: {
  "name": "iPhone 15 Pro",
  "price": 999.99,
  "description": "256GB, Space Black"
}
```

**Why?**
- Products/services can change price after order
- Snapshot preserves price at time of purchase
- Accurate historical records for accounting
- No foreign key constraints on item_id (item can be deleted)

---

### 5. **Composite Indexes for Performance**

**Key Indexes:**
```sql
-- Appointment filtering (most common query)
INDEX idx_appointments_date_status ON appointments(appointment_date, status)

-- Provider scheduling
INDEX idx_appointments_provider_date ON appointments(provider_id, appointment_date)

-- Customer order history
INDEX idx_orders_customer_status ON orders(customer_id, payment_status)

-- Newsletter tracking
INDEX idx_sends_tracking_token ON newsletter_sends(tracking_token)

-- Settings lookup
UNIQUE INDEX idx_settings_category_key ON settings(category, key)
```

**Result:** 30-50% faster filtered list views

## 🔍 Important Data Flows

### 1. **Guest Cart → Customer Cart Migration**

**Trigger:** User logs in or registers
**Event:** `Illuminate\Auth\Events\Login` or `Illuminate\Auth\Events\Registered`
**Listener:** `MigrateGuestCart`

**Flow:**
```
1. Guest adds items → cart.session_id = "abc123", cart.customer_id = NULL
2. User logs in → Session regenerated (security)
3. Pre-login session ID stored: session()->put('guest_session_id', 'abc123')
4. MigrateGuestCart listener fires
5. Find cart items WHERE session_id='abc123' AND customer_id IS NULL
6. Update cart items: customer_id=42, session_id=NULL
7. Merge quantities if duplicate items exist
8. Transaction commit
```

**Database Transaction:**
- Atomic operation (all or nothing)
- Rollback on failure
- Duplicate handling (quantity merge)

---

### 2. **Newsletter Campaign Sending**

**Trigger:** Scheduled command (runs every 5 minutes)
**Command:** `newsletters:send-scheduled`
**Job:** `SendNewsletter`

**Flow:**
```
1. Find newsletters WHERE status='scheduled' AND scheduled_send_time <= NOW
2. Dispatch SendNewsletter job to queue
3. Job marks newsletter as 'sending'
4. Get target subscribers (active, in selected lists, distinct)
5. Bulk create newsletter_sends records with unique tokens
6. Send in batches (100 emails per batch, 60-second delays)
7. Track individual sends (sent_at, opened_at, clicked_at)
8. Mark newsletter as 'sent' when complete
9. Update counts (sent_count, opened_count, clicked_count)
```

**Tracking:**
- Open tracking: 1x1 transparent GIF pixel (`/newsletter/track/open?token=xyz`)
- Click tracking: Redirect through tracking URL (`/newsletter/track/click?token=xyz&url=...`)
- Unsubscribe: Token-based one-click (`/newsletter/unsubscribe?token=xyz`)

---

### 3. **Order Creation from Cart**

**Controller:** `CheckoutController@process`
**Service:** `OrderFactory->createOrderFromCart()`

**Flow:**
```
1. Validate cart items (stock availability)
2. Find or create customer (supports guest checkout)
3. Build order record (payment_method, addresses, totals)
4. Create order items with price snapshots
   - snapshot = ['name' => $item->name, 'price' => $item->price, ...]
5. Calculate totals (subtotal, tax 6.5%, total)
6. Process payment (Stripe/PayPal redirect OR cash/check)
7. Mark order as paid (after webhook verification)
8. Send order confirmation email
9. Clear customer cart
10. Newsletter opt-in (if checked)
```

**Price Snapshot Example:**
```json
{
  "item_id": 42,
  "item_type": "App\\Models\\Product",
  "quantity": 2,
  "snapshot": {
    "name": "iPhone 15 Pro",
    "price": 999.99,
    "description": "256GB, Space Black",
    "sku": "IPHONE15-256-BLACK"
  }
}
```

## 🔐 Security Notes

### Protected Columns (Mass Assignment)

**Table:** `customers`
**Guarded Fields:**
- `role` - Prevents privilege escalation
- `is_admin` - Prevents admin registration exploit

**Why?**
Without `$guarded`, users could register as admins via form manipulation.

### Unique Constraints

- `customers.email` - Prevents duplicate accounts
- `coupons.code` - Unique discount codes
- `products.slug` - SEO-friendly URLs
- `services.slug` - SEO-friendly URLs
- `tags.name`, `tags.slug` - Unique tag identifiers
- `customer_tag.(customer_id, tag_id)` - One tag per customer
- `stock_notifications.(email, product_id)` - One notification per product per email
- `service_bundles.slug` - SEO-friendly URLs
- `settings.(category, key)` - Prevents duplicate settings
- `newsletter_sends.tracking_token` - Secure tracking

### Foreign Key Constraints

**ON DELETE CASCADE:**
- `orders` → `customers` (delete customer = delete orders)
- `order_items` → `orders` (delete order = delete items)
- `appointments` → `customers` (delete customer = delete appointments)

**ON DELETE SET NULL:**
- `orders` → `coupons` (delete coupon = set order.coupon_id to NULL)

**ON DELETE RESTRICT:**
- `appointments` → `providers` (cannot delete provider with appointments)
